## 二、核心写入函数深度分析

### 2.1 `handleStockUpdate` — 存在严重闭包陷阱

**文件位置**：`src/App.tsx:252-271`

```typescript
const handleStockUpdate = (productId: string, amount: number) => {
  if (isHistoryMode) return;
  
  // 第一处更新：products (使用函数式更新，正确 ✓)
  setProducts(prev => prev.map(product => 
    product.id === productId 
      ? { ...product, stock: Math.min(product.stock + amount, product.maxStock), outOfStockRegistered: false }
      : product
  ));

  // 第二处更新：schedule (使用函数式更新，正确 ✓)
  setSchedule(prev => prev.map(item => 
    item.productId === productId && item.status !== 'completed'
      ? { ...item, status: 'completed' as const }
      : item
  ));

  // ⚠️ 第三处更新：selectedProduct (使用闭包变量，错误 ✗)
  const updatedProduct = products.find(p => p.id === productId);  // ❌ 引用的是旧闭包值！
  if (updatedProduct) {
    const newStock = Math.min(updatedProduct.stock + amount, updatedProduct.maxStock);
    setSelectedProduct({ ...updatedProduct, stock: newStock, outOfStockRegistered: false });
  }
};
```

### 2.2 问题根因 — React 闭包陷阱详解

| 问题类型 | 具体表现 | 影响程度 |
|---------|---------|---------|
| **Stale Closure** | 第266行 `products.find()` 读取的是函数定义时捕获的 `products` 快照，而非 `setProducts` 刚更新后的最新状态 | 🔴 严重 |
| **状态不同步** | `products` 与 `selectedProduct` 可能存储不同的 `stock` 值，导致面板与计算器展示不一致 | 🔴 严重 |
| **多次调用叠加** | 短时间内连续调用会基于同一旧值计算新库存，导致最终写入量丢失 | 🟠 高 |

#### 场景复现：

```
初始状态：products[0].stock = 10, selectedProduct.stock = 10

T1: 调用 handleStockUpdate(id, 5)
    → setProducts(prev => prev.map(...))   // products 将变为 15
    → products.find() 仍然返回 stock=10    // 闭包旧值！
    → setSelectedProduct(stock=10+5=15)    // selectedProduct = 15

T2: 紧接着再调用 handleStockUpdate(id, 3)  ← React 批处理周期内
    → setProducts(prev => prev.map(...))   // products 将变为 18 (正确: 15+3)
    → products.find() 仍然返回 stock=10    // 还是旧闭包！
    → setSelectedProduct(stock=10+3=13)    // selectedProduct = 13 ❌ 错了！

最终结果：
  products[0].stock = 18  ✓
  selectedProduct.stock = 13  ✗  丢失了第一次的5件补货！
```

### 3.5 入口5：快速补货 `handleQuickRestock` — 双重计算不一致

**文件位置**：`src/App.tsx:294-301`

```typescript
const handleQuickRestock = (productId: string) => {
  if (isHistoryMode) return;
  const product = products.find(p => p.id === productId);  // ❌ 也用了旧闭包值！
  if (!product) return;
  const amount = product.maxStock - product.stock;         // ❌ 基于旧 stock 计算
  if (amount <= 0) return;

```

**风险场景**：
```

T1: 用户点击计算器「一键补货」
    → handleQuickRestock 计算 amount = 30 - 5 = 25 ✓
    → handleStockUpdate(id, 25) 正在处理...
    
T2: React 批处理未完成前，用户又在商品面板点击「一键补货」
    → products.find() 还是返回 stock=5 (旧闭包)
    → 再次计算 amount = 30 - 5 = 25 ❌ 实际只需补 0 件！
    → handleStockUpdate(id, 25) 再次被调用
    
最终结果：
  products: setProducts 函数式更新会正确处理，最终为 30 ✓
  selectedProduct: 第一次 setSelectedProduct(30)，第二次 setSelectedProduct(30) 可能也对
  BUT: 第二次调用 handleStockUpdate 内部 setProducts(prev => stock+25) 会变成 30+25=55 被 Math.min 截断为 30
  → 虽然有 Math.min 保护不超过 maxStock，但逻辑语义完全错误！
```

### 4.2 三大断裂场景

#### 断裂场景一：补货后 selectedProduct 未同步（核心BUG）

| 时间点 | 操作 | products[id].stock | selectedProduct.stock | 面板显示 | 计算器显示 | 一致？ |
|-------|------|-------------------|----------------------|---------|----------|-------|
| T0 | 初始 | 10 | 10 | 10/30 | 10/30 | ✅ |
| T1 | 计算器点击「确认补货」+5 | 15 | 15 (同批次更新) | 15/30 | 15/30 | ✅ |
| T2 | 快速连续点击再补 +3 | 18 | **13** (旧闭包 10+3) | 18/30 | **13/30** | ❌ **断裂** |
| T3 | 在商品面板点击其他商品再点回来 | 18 | 18 (从 products 重新读取) | 18/30 | 18/30 | ✅ 恢复 |

#### 断裂场景二：预测面板补货后双面板不一致

| 时间点 | 操作 | products[id].stock | selectedProduct.stock | 面板显示 | 计算器显示 |
|-------|------|-------------------|----------------------|---------|----------|
| T0 | 商品 A 被选中，stock=8 | 8 | 8 | 8/20 | 8/20 |
| T1 | 切到预测面板，点击对 A「一键补货」+12 | 20 | **8** ❌ (预测面板不走 setSelectedProduct 更新路径！) |
| T2 | 切回补货排程 Tab | 20 | **8** | **20/20 充足** | **8/20 建议补12** ❌ 完全矛盾！ |

**根因**：`ReplenishmentForecast` 的 `onReplenish` 只调用了 `handleStockUpdate`，而 `handleStockUpdate` 中 `setSelectedProduct` 的前提是 `products.find()` 找到该商品（能找到），但用的是旧闭包值——所以即使在 T1 执行时，`selectedProduct` 也只会被设置为 `8+12=20`（本例中恰好正确，但如果之前有操作就会错）。更严重的是：**如果用户在预测面板补货时，该商品根本不是 selectedProduct，则 `setSelectedProduct` 完全不会被触发！**

#### 断裂场景三：送货核验批量更新后最后一个商品「污染」

| 步骤 | 送货商品 | 原库存 | 补货量 | 补货后 products.stock | 补货后 selectedProduct |
|-----|---------|-------|-------|---------------------|----------------------|
| 初始选中商品 C | C | 5 | - | 5 | C.stock=5 |
| 核验送货单（含A、B、C） | A | 10 | +20 | 30 | A.stock=30 ❗ (被覆盖) |
| | B | 8 | +15 | 23 | B.stock=23 ❗ (被覆盖) |
| | C | 5 | +10 | 15 | C.stock=15 ✓ (最后一次) |

如果顺序反过来（C、A、B），最后 selectedProduct 会变成 B 的数据，而用户选中的是 C！

**修复后代码**：
```typescript
const handleStockUpdate = (productId: string, amount: number) => {
  if (isHistoryMode) return;
  
  // 使用函数式更新 + 在同一个 setProducts 回调中同步 selectedProduct
  setProducts(prev => {
    const newProducts = prev.map(product => 
      product.id === productId 
        ? { ...product, stock: Math.min(product.stock + amount, product.maxStock), outOfStockRegistered: false }
        : product
    );
    
    // 在同一个闭包中使用最新值更新 selectedProduct
    const updatedProduct = newProducts.find(p => p.id === productId);
    if (updatedProduct) {
      // 使用 queueMicrotask 确保状态更新顺序（或直接用函数式）
      queueMicrotask(() => {
        setSelectedProduct(currentSel => {
          if (!currentSel || currentSel.id !== productId) return currentSel;
          return { ...currentSel, stock: updatedProduct.stock, outOfStockRegistered: false };
        });
      });
    }
    
    return newProducts;
  });

  setSchedule(prev => prev.map(item => 
    item.productId === productId && item.status !== 'completed'
      ? { ...item, status: 'completed' as const }
      : item
  ));
};
```

### 6.2 架构优化 — 消除 selectedProduct 冗余状态

**根本问题**：`selectedProduct` 是 `products` 的**派生数据**，不应该用独立 `useState` 存储！

**推荐方案：改为「选择ID + useMemo 派生」模式**

```typescript
// 在 App.tsx 中：
// 原：const [selectedProduct, setSelectedProduct] = useState<Product | null>(null);
// 改为：
const [selectedProductId, setSelectedProductId] = useState<string | null>(null);

const selectedProduct = useMemo(() => {
  if (!selectedProductId) return null;
  return displayProducts.find(p => p.id === selectedProductId) || null;
}, [displayProducts, selectedProductId]);

const handleProductSelect = (product: Product) => {
  setSelectedProductId(product.id);  // 只存 ID，不存对象副本
};
```

**好处**：
- ✅ `selectedProduct` 始终与 `displayProducts`（即 `products` 或快照）保持绝对同步
- ✅ 无需在每次库存更新后手动 `setSelectedProduct`
- ✅ 消除了所有 `setSelectedProduct` 带来的竞态和闭包问题
- ✅ 代码量减少，bug 面收窄

### 6.3 预测面板补货 — 增加实时库存校验

**在 `ReplenishmentForecast` 的 `handleReplenish` 之前或在 `App.handleStockUpdate` 内部：**

```typescript
// 方案一：App.handleStockUpdate 中增加校验（推荐，拦截所有入口）
const handleStockUpdate = (productId: string, amount: number) => {
  if (isHistoryMode) return;
  
  setProducts(prev => {
    const product = prev.find(p => p.id === productId);
    if (!product) return prev;
    
    // 实际可补数量 = max(0, maxStock - stock)
    const actualCanReplenish = Math.max(0, product.maxStock - product.stock);
    // 按最小的来，避免超额补货
    const actualAmount = Math.min(amount, actualCanReplenish);
    
    if (actualAmount <= 0) return prev;  // 无需补货，直接跳过
    
    return prev.map(p => 
      p.id === productId 
        ? { ...p, stock: p.stock + actualAmount, outOfStockRegistered: false }
        : p
    );
  });
  // ... 其余同上
};
```

### 6.4 handleQuickRestock 同步修复

**修复后**：
```typescript
const handleQuickRestock = (productId: string) => {
  if (isHistoryMode) return;
  // 不在外部计算 amount，交给 handleStockUpdate 内部（函数式更新内）做校验计算
  // 或者直接补一个超大值，让 handleStockUpdate 内部的 Math.min 保护生效
  const product = displayProducts.find(p => p.id === productId);  // 改用 displayProducts
  if (!product) return;
  const amount = product.maxStock - product.stock;
  if (amount <= 0) return;
  handleStockUpdate(productId, amount);
};
```
